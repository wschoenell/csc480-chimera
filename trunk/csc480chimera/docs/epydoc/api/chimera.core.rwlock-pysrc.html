 21      """Read-Write lock class. A read-write lock differs from a standard 
 22      threading.RLock() by allowing multiple threads to simultaneously hold a 
 23      read lock, while allowing only a single thread to hold a write lock at the 
 24      same point of time. 
 25   
 26      When a read lock is requested while a write lock is held, the reader 
 27      is blocked; when a write lock is requested while another write lock is 
 28      held or there are read locks, the writer is blocked. 
 29   
 30      Writers are always preferred by this implementation: if there are blocked 
 31      threads waiting for a write lock, current readers may request more read 
 32      locks (which they eventually should free, as they starve the waiting 
 33      writers otherwise), but a new thread requesting a read lock will not 
 34      be granted one, and block. This might mean starvation for readers if 
 35      two writer threads interweave their calls to acquireWrite() without 
 36      leaving a window only for readers. 
 37   
 38      In case a current reader requests a write lock, this can and will be 
 39      satisfied without giving up the read locks first, but, only one thread 
 40      may perform this kind of lock upgrade, as a deadlock would otherwise 
 41      occur. After the write lock has been granted, the thread will hold a 
 42      full write lock, and not be downgraded after the upgrading call to 
 43      acquireWrite() has been match by a corresponding release(). 
 44      """
